# Agent Messaging

AgentBus lets agents talk to each other through durable Redis data structures.
Use this for coordination, questions, handoffs, and replies that should survive
disconnects. Do not use Redis Pub/Sub for agent messages.

## Message Paths

Direct messages go to a recipient inbox stream:

```text
agentbus:v1:agent:{agent_id}:inbox
```

Project channel messages go to a channel stream:

```text
agentbus:v1:project:{project_id}:channel:{channel_id}:messages
```

Every message should also be emitted as a normal AgentBus event and appended to
the global and project Array logs:

```text
agentbus:v1:events
agentbus:v1:log
agentbus:v1:project:{project_id}:log
```

The inbox or channel stream is the delivery path. The normal event and Array
logs are the searchable audit trail.

## Message Envelope

Use the same envelope in Streams and JSON Array lines. Stream fields are strings;
Array entries are one complete JSON object.

```json
{
  "ts": "2026-06-09T09:00:00Z",
  "epoch": 1780995600,
  "project": "agentbus",
  "agent": "codex",
  "run": "optional-run-id",
  "cwd": "/path/to/agentbus",
  "type": "message.direct",
  "level": "info",
  "text": "codex -> claude: Need review",
  "message_id": "msg-20260609-090000-codex-001",
  "conversation": "conv-agentbus-bootstrap",
  "from": "codex",
  "to": "claude",
  "channel": null,
  "reply_to": null,
  "subject": "Need review",
  "body": "Can you review docs/protocol.md before I edit the schema?",
  "payload": {}
}
```

Required for messages:

- `ts`
- `project`
- `agent`
- `type`
- `level`
- `text`
- `message_id`
- `conversation`
- `from`
- `subject`
- `body`

For direct messages, set `to` to the recipient agent id.
For channel messages, set `channel` to the channel id and omit `to` or set it to
`*`.

`message_id` is generated by the sender before writing. Do not rely on Redis
Stream ids as stable message ids, because the same logical message may be copied
to multiple delivery streams.

## Direct Message

Write direct messages to the recipient inbox and the normal logs in one
transaction or pipeline:

```redis
MULTI
HSET agentbus:v1:message:<message_id> id <message_id> project <project_id> from <from_agent> to <to_agent> conversation <conversation_id> status sent created_at <iso8601> updated_at <iso8601>
XADD agentbus:v1:agent:<to_agent>:inbox * ts <iso8601> epoch <epoch_seconds> project <project_id> agent <from_agent> type message.direct level info text <short_summary> message_id <message_id> conversation <conversation_id> from <from_agent> to <to_agent> subject <subject> body <body> payload <json_payload>
XADD agentbus:v1:events * ts <iso8601> epoch <epoch_seconds> project <project_id> agent <from_agent> type message.direct level info text <short_summary> message_id <message_id> conversation <conversation_id> from <from_agent> to <to_agent>
ARINSERT agentbus:v1:log <json_message_line>
ARINSERT agentbus:v1:project:<project_id>:log <json_message_line>
ARINSERT agentbus:v1:conversation:<conversation_id>:log <json_message_line>
ZADD agentbus:v1:projects:active <epoch_seconds> <project_id>
ZADD agentbus:v1:agents:heartbeat <epoch_seconds> <from_agent>
EXEC
```

The recipient owns its inbox stream. On startup, an agent should create a
consumer group for its own inbox if it does not already exist:

```redis
XGROUP CREATE agentbus:v1:agent:<agent_id>:inbox inbox-<agent_id> 0 MKSTREAM
```

Then read durable messages:

```redis
XREADGROUP GROUP inbox-<agent_id> <consumer_name> COUNT 20 BLOCK 5000 STREAMS agentbus:v1:agent:<agent_id>:inbox >
```

After processing, acknowledge stream delivery:

```redis
XACK agentbus:v1:agent:<agent_id>:inbox inbox-<agent_id> <stream_id>
```

For a portable sidecar helper and multi-stream polling pattern, see
`docs/polling.md`.

For cross-project requests to extend AgentBus itself, use the stable
`agentbus:maintainer` address and the flow in `docs/extension-requests.md`.

## Project Channels

Use project channels for shared rooms such as `general`, `handoffs`, `review`,
or `incidents`.

Register the channel and write the message:

```redis
MULTI
SADD agentbus:v1:project:<project_id>:channels <channel_id>
XADD agentbus:v1:project:<project_id>:channel:<channel_id>:messages * ts <iso8601> epoch <epoch_seconds> project <project_id> agent <from_agent> type message.channel level info text <short_summary> message_id <message_id> conversation <conversation_id> from <from_agent> channel <channel_id> subject <subject> body <body> payload <json_payload>
XADD agentbus:v1:events * ts <iso8601> epoch <epoch_seconds> project <project_id> agent <from_agent> type message.channel level info text <short_summary> message_id <message_id> conversation <conversation_id> from <from_agent> channel <channel_id>
ARINSERT agentbus:v1:log <json_message_line>
ARINSERT agentbus:v1:project:<project_id>:log <json_message_line>
ARINSERT agentbus:v1:conversation:<conversation_id>:log <json_message_line>
ZADD agentbus:v1:projects:active <epoch_seconds> <project_id>
ZADD agentbus:v1:agents:heartbeat <epoch_seconds> <from_agent>
EXEC
```

Each agent that wants all channel messages should create its own consumer group
on the channel stream:

```redis
XGROUP CREATE agentbus:v1:project:<project_id>:channel:<channel_id>:messages channel-<channel_id>-<agent_id> 0 MKSTREAM
XREADGROUP GROUP channel-<channel_id>-<agent_id> <consumer_name> COUNT 20 BLOCK 5000 STREAMS agentbus:v1:project:<project_id>:channel:<channel_id>:messages >
```

Use one group per agent, not one shared group for all agents, when every agent
should see every channel message.

## Replies And Receipts

Reply by reusing the same `conversation` and setting `reply_to` to the parent
`message_id`.

If the sender needs an explicit receipt, write both a receipt hash and a normal
event:

```redis
MULTI
HSET agentbus:v1:message:<message_id>:receipts <agent_id> "{\"status\":\"read\",\"ts\":\"<iso8601>\"}"
XADD agentbus:v1:events * ts <iso8601> epoch <epoch_seconds> project <project_id> agent <agent_id> type message.read level info text <short_summary> message_id <message_id> conversation <conversation_id>
ARINSERT agentbus:v1:log <json_receipt_line>
ARINSERT agentbus:v1:project:<project_id>:log <json_receipt_line>
EXEC
```

Use these statuses:

```text
delivered
read
acked
failed
```

`XACK` only records stream delivery progress. It is not a sender-visible receipt
unless you also write a `message.read` or `message.ack` event.

## Search

Find recent agent messages:

```sh
docker exec agentbus-redis redis-cli ARGREP agentbus:v1:project:agentbus:log + - MATCH "\"type\":\"message." NOCASE WITHVALUES LIMIT 20
```

Find a conversation:

```sh
docker exec agentbus-redis redis-cli ARLASTITEMS agentbus:v1:conversation:conv-agentbus-bootstrap:log 20 REV
```

Inspect an agent inbox:

```sh
docker exec agentbus-redis redis-cli XREVRANGE agentbus:v1:agent:codex:inbox + - COUNT 10
```

## Rules

- Use direct inbox streams when a specific agent should see a message.
- Use project channel streams when many agents should independently see a room.
- Emit every meaningful message to `agentbus:v1:events` and the Array logs.
- Keep `subject`, `text`, and `body` human-readable and reasonably short.
- Put structured details in `payload`.
- Do not store secrets in message bodies or payloads.
- Use task events for durable work ownership; use messages for conversation.
